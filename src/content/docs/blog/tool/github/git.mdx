既然 Github 能够存储代码，那么我们怎么把本地的代码上传到 Github 呢？这就需要用到 Git 了。

那什么又是 Git 呢？

## 1. Git 简介

Git 是一个“分布式版本控制系统”，是一款专业的让你可以用“后悔药” + “分身术”的工具。

- **后悔药**：就是版本控制。比如：你写博客代码，今天改了 A，明天改了 B，后天发现还是 A 好。Git 能让你随时一键“穿越”回任何一个历史时刻。
- **分身术**：就是分布式。每个人的电脑上都有一个完整的代码库副本。即使断网了，你也能在本地提交代码；等有网了，再同步到 GitHub。
- **协作**：它能把多个人对同一个文件的修改，像织毛衣一样安全地缝合在一起。

## 3. Git CLI（命令行）工具

为什么有了 GUI，还要学命令行（CLI）呢？这是很多新手最困惑的地方。既然点点鼠标就能完成，为什么要背那些难记的指令（如 `git rebase -i`）？

我觉得有这 4 个理由：

- **极客的“肌肉记忆”**：熟练之后，键盘输入 `git push` 的速度远快于：`伸手拿鼠标 -> 找到按钮 -> 点击确认`。在高手眼里，双手不离开键盘才是最高效的状态。
- **自动化与远程服务器**：当你以后写博客涉及到 AI 部署或 CI/CD 自动化时：
  - 服务器（如 Linux 云主机）是没有图形界面的，你只能通过命令行操作。
  - 自动化脚本（GitHub Actions）也只能识别命令，它没法帮你去点击 SourceTree 上的按钮。
- **稳定性与性能**：GUI 工具本质上是在后台帮你在跑命令行。有时候 GUI 会卡死、报错信息模糊，或者在大规模仓库下反应迟钝。命令行则是直接与内核对话，最快、最稳、报错最准。
- **命令行是“完整版”，GUI 是“精简版”**：Git 有几百个功能指令，GUI 工具为了简洁，通常只把最常用的 20% 做成按钮。比如：当你遇到复杂的代码冲突，或者需要“捡樱桃”（cherry-pick）特定的代码提交，甚至要把几个提交记录合并成一个（squash）时，GUI 往往束手无策，只能靠命令行救命。

## 4. Git 常用指令

我个人觉得掌握一套“肌肉记忆”级别的 Git 指令是必须的。

我们可以按照你从新建项目到日常提交，再到解决麻烦的顺序来梳理：

### 4.1 建立联系（初始化与下载）

当你开始一个新博文的 Demo 或新项目时：

- `git init`：在当前文件夹新建一个本地仓库。
- `git clone [url]`：把 GitHub 上的项目“下载”到本地。
- `git remote add origin [url]`：把本地仓库和 GitHub 上的远程仓库关联起来。

### 4.2 日常三部曲（最常用）

这三个指令是你写博客代码时，每天重复 100 遍的操作：

- `git add .`：把所有修改过的文件“暂存”起来（准备提交）。
- `git commit -m "描述信息"`：给这次修改打个包，并写上备注（比如 "feat: 完成登录页面"）。
- `git push origin [分支名]`：把代码正式推送到 GitHub 上。


### 4.3 状态与对比（看你在干嘛）

- `git status`：最常用。查看哪些文件改了还没提交，哪些文件漏了。
- `git log`：查看提交历史，看看你之前都干了啥。
- `git diff`：查看具体代码行级别的改动。

### 4.4 分支管理（多任务并行）

比如你正在写“前端”代码，突然想去改下“后端”，可以用分支隔开：

- `git branch`：查看当前有哪些分支。
- `git checkout -b [分支名]`：创建一个新分支并直接跳过去。
- `git merge [分支名]`：把那个分支的代码合并到当前分支。

### 4.5 救急与撤销（反悔药）

如果你不小心写乱了：

- `git pull`：把远程最新的代码拉下来，覆盖/更新本地。
- `git checkout [文件名]`：撤销某个文件还没提交的修改。
- `git reset --hard HEAD^`：慎用。回退到上一个版本，彻底抹掉当前的改动。

### 给博主的专业建议

在 GitHub 上写博文示例时，建议养成 "原子化提交" 的习惯：

- **不要**写了一整天代码才 Commit 一次。
- **要**实现一个具体小功能（比如加了个按钮）就 Commit 一次。

这样你的 GitHub 贡献墙（绿格子）会更漂亮，代码回退也更方便。
